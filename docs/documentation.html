Documentation

RE-Argo の CLI オペレーション・リファレンス。すべてのコマンド・フラグ・出力規約を、 LLM・人間どちらでも試行錯誤なしに使えるよう記述しています。これはリポジトリの CLAUDE.md を正として生成された HTML 版です。

CLI の呼び出し方

# 開発ビルド
cargo run --release -- <command> <args>

# リリースビルド後
target/release/re-argo <command> <args>

16 進アドレスは 0x401000 / 401000 / 0X401000 のいずれでも可。 グローバルフラグ --thumb は 32-bit ARM を Thumb モードでデコードする場合のみ使用します （x86_64 / aarch64 では不要）。

ビルド / テスト / Lint

cargo build              # debug build
cargo build --release    # release build
cargo test --workspace   # 約 600 テスト
cargo clippy --workspace --all-targets -- -D warnings

探索 / トリアージ

info discovery

形式・アーキテクチャ・ビット幅・エントリポイント・セクション数・シンボル数に加え、検出できればコンパイラ / ランタイムのフィンガープリント、高シグナルなトリアージ値（imphash / packer / entropy_overall）を表示。出力キー例: Format, Architecture, language, runtime, libc_version, imphash, tlsh, richhash, signed, packer。

triage 最初の一手

1 画面のマルウェアトリアージレポート。フルパイプラインを実行し、形式 / アーキ / エントリ、識別子（コンパイラ / 言語 / ランタイム）、ハッシュ（imphash / tlsh）、コード署名情報、パッカー + 全体エントロピー、capa ルールマッチ（上位 10）、CWE 所見、IoC、タグ数をまとめて表示。「これは何で、心配すべきか？」を一発で。

summary

1 画面のダイジェスト: 形式 / エントリ / ランタイム識別 / 関数数（総数・命名率・no-return・thunk）/ 注釈密度 / fan-in 上位関数 / 再帰クラスタ数 / トリアージブロック（packer, entropy, imphash, capa, CWE, IoC 数）/ kind 別タグ内訳。

sections / memmap

sections は Name / Address / Size / Flags の表。memmap は同じデータをアドレス順に並べ、log₂ スケールのサイズバー付きで表示。

symbols / imports / strings

symbols <FILE> [-k func|data|import|export] [--all] [-n SUBSTR] — 既定はローダ側シンボル（高速）。ストリップ済みバイナリでは --all でフル解析を走らせ、RTTI / VTable / アナライザー復元名を含めます。

imports <FILE> [--exports] — imports（既定）/ exports を列挙。strings <FILE> [-m N] [--all] — 印字可能文字列（既定最小長 4）。

解析パイプライン

analyze

フルの 68 アナライザーパイプラインを実行。多くのコマンドが内部で暗黙に実行するため、analyze は主に (1) 各アナライザーの寄与を stderr で確認、(2) パイプラインのみを走らせる、用途で使います。

functions / metrics

functions — 発見した全関数（アドレス / 名前 / ブロック数）。metrics <FILE> [--sort COL] [--top N] — 関数別の McCabe / blocks / insns / fan-in / fan-out / stack。ソット列: mccabe(既定) / blocks / insns / fan_in / fan_out / stack。

coverage / bench / workflow

coverage [--annotated] — 解析がデコードした実行可能バイトの割合。bench — 解析のウォールクロック + ステージ別内訳。workflow [--declared-only] — 全アナライザーを優先度順に provides → consumes 付きで列挙し、最後にパイプラインを検証。

所見 + 分類

tags BN スタイル

tags <FILE> [--filter KIND] [--list-types] — 分類済みのアナライザー所見。--list-types で kind 別カウント要約。--filter の kind スラッグ（カノニカル）: crypto, suspicious, library, anti_debug, noreturn, wrapper, important, loop, recursive, stack_protected, exception, bug, crash。出力: scope kind address auto text。

find

find <FILE> <QUERY> [--limit N] — 関数 / シンボル / コメント / タグを大文字小文字無視の部分一致で横断検索。1 ショットで名前を見つける最速手段。

signatures

signatures [--library LIB] [--export PATH] — 組み込みシグネチャ DB（約 312 エントリ、libc / Win32 / POSIX / libstdc++）の閲覧 / エクスポート。

コード閲覧

disasm / pcode

disasm <FILE> [-s ADDR] [-n N] — ADDR から N 命令を逆アセンブル（既定はエントリ）。pcode <FILE> [-s ADDR] [-n N] — リフト済み P-code IR を表示。

decompile / decompile-all

decompile <FILE> [-a ADDR] [--ssa] [--rust] — 関数を疑似コードへ。--ssa で最適化済み SSA IR、--rust で Rust 風出力。インライン // <comment> 注釈とコールレンダリング置換（printf("hi", 42)）を含みます。decompile-all [-o DIR] [--skip-errors] で全関数を並列に。

cfg

cfg <FILE> <ADDRESS> — 単一関数の基本ブロック CFG を Graphviz DOT で出力。dot -Tpng へパイプ可。

参照 / 呼び出し関係

xrefs / callgraph

xrefs <FILE> <ADDRESS> — そのアドレスへの / からの相互参照（TYPE: CALL / JUMP / READ / WRITE …）。callgraph [--dot] [--scc] [--around ADDR] [--depth N] — コールグラフ。--scc で強連結成分（相互 / 直接再帰）。

backtrace / callsites

backtrace <FILE> <ADDRESS> [--depth N] [--max-paths N] — ADDRESS へ至る逆コールグラフの全経路（最短優先・循環検出）。既定 depth 6。callsites [-t ADDR] [-n N] [--callbacks] — 静的に解決した引数値つきの全コールサイト。

能力検出 / トリアージ層

entropy

セクション別 Shannon エントロピー（bits/byte, 0.0–8.0）。7.0 超は high entropy、7.5 超は likely packed とフラグ。パッカー疑いサンプルの第一手。

capa / packer

capa [--namespace SUBSTR] — Capa スタイルの能力レポート。組み込みルールが imports / strings / tags にマッチし、host-interaction/file-system/read-file のような namespace + 名前を返します。packer — DIE / PEiD スタイルのパッカー検出（UPX / ASPack / Themida / VMProtect …）。

ioc / vuln

ioc [--kind KIND] — IoC 抽出。全文字列を url / ipv4 / ipv6 / email / registry-key / named-pipe / mutex / win-path / posix-path / eth-addr / btc-addr / user-agent / domain に分類。vuln — Cwe_checker スタイルの脆弱パターン（CWE-78 / -120 / -134 / -242 / -330 / -426 / -676）。

yara / floss

yara <FILE> <RULES.yar> [--sample N] — YARA-lite スキャン（テキスト / ワイルドカード hex / ブール条件 / N of them）。非対応構文はパース時に明示エラー。floss [--min-length N] [--with-add] [--limit N] — XOR / ROL / ADD 難読化文字列のブルートフォースデコード。

rop / embedded

rop [--kinds rop|jop|cop|all] [--useful-only] [--contains TEXT] [--limit N] — x86 / x64 の ROP / JOP / COP ガジェット探索（ROPgadget / ropper 形式）。embedded [--limit N] — Binwalk スタイルの埋め込みファイル走査（ELF / PE / ZIP / PNG / PDF / SQLite …）。

tlsh-diff

tlsh-diff <A> <B> — TLSH（内容ファジーハッシュ）で 2 バイナリを比較。距離 < 30 で同一ファミリの可能性、< 50 で関連、> 100 で無関係。

データ / 検索

hexdump / search

hexdump <FILE> <ADDRESS> [LENGTH] — アドレスから 16 バイト幅で hex ダンプ（既定 256 バイト）。search [--hex PAT] [--text PAT] [-n N] — バイト / テキスト検索。--hex "48 8b ?? 24" は ?? ワイルドカード、--text は正規表現。

Diff

diff / semantic-diff

diff <A> <B> — バイト / シンボルテーブルレベルの比較。semantic-diff <A> <B> [-k KINDS] — SSA / 逆コンパイル関数レベルで identical / renamed / tweaked / modified / added / removed を報告。

エクスポート

export / export-xml

export <FILE> [-o PATH] — 解析の JSON ダンプ。export-xml <FILE> [-o PATH] — Ghidra 互換 XML。JSON のトップレベルキーは name / format / arch / functions[] / symbols[] / references[] / tags{} ほか。

エミュレーション / デバッグ

emulate / gdbserver / debug

emulate [-s ADDR] [-n STEPS] [-b ADDR …] — P-code エミュレータを ADDR から最大 STEPS ステップ実行（決定的）。gdbserver [-l HOST:PORT] — GDB RSP サーバ（既定 127.0.0.1:1234、gdb-multiarch / lldb 互換）。debug — 対話 REPL。

手動補正レイヤー

annotate / iterate / carve

annotate [--rename ADDR=NAME] [--func ADDR] [--comment ADDR=TEXT] [--import FILE] [--list] [--clear] — 永続オーバーライド・サイドカー <binary>.gra.json を編集し、以後の解析へ再適用。iterate [--apply] [--max-rounds N] — 自動補正の不動点ループ。carve [--address ADDR | --start --end | --start --size] [--format elf|raw] [-o OUT] — 単一関数 / VA 範囲を小さな単体ファイルへ切り出し（元の VA を維持）。GUI Ghidra が読めない巨大バイナリの一部だけを解析する用途。

詳細な carve の設計メモは docs/carve-command.md を参照。

スクリプト / 統合

script / mcp / taint / patch

script <FILE> <SCRIPT.grs> — .grs スクリプトから複数コマンドを実行。mcp <FILE> — Model Context Protocol を stdio で話し、AI ホスト（Claude Code 等）へ埋め込み。公開ツール: list_functions, decompile_function, disassemble, find_xrefs, list_symbols, get_program_info。taint [-a ADDR] [-p N] — N 個の引数レジスタをテイントし SSA 経由で危険なシンクへ伝播。patch <ADDR> [--bytes | --asm] [-o OUT] — バイナリのパッチ。

出力フォーマット規約

  • アドレス列: 小文字 16 進 + 0x プレフィックス（0x401234）。狭い列ではゼロ拡張しない。
  • 関数参照: name (0xADDR) 形式（例 printf@plt (0x401050)）。シンボルなしは FUN_<addr>、PLT サンクは @plt サフィックス。
  • タグの scope: address（命令 / データバイト）または function（エントリ）。組み込み kind スラッグ: important, suspicious, library, crypto, bug, crash, noreturn, wrapper, recursive, loop, stack_protected, anti_debug, exception。
  • JSON (export): 各関数は {address, name, block_count, call_count, stack_size}、各シンボルは {address, name, kind}（kind ∈ Function / ExternalFunction / Data / Label）。

よくある落とし穴

  1. <ADDRESS> 引数は 16 進必須（0x401000 または 401000、4198400 ではない）。
  2. symbols は既定でローダ側のみ。ストリップ済みバイナリでは --all を付ける。
  3. cargo run は毎回リビルド。cargo build --release 後に target/release/re-argo を直接呼ぶと速い（.gra.json サイドカー以外に永続キャッシュはなし）。
  4. --thumb は 32-bit ARM のみ。x86_64 / aarch64 では付けない。
  5. find <bin> <query> は大文字小文字無視の部分一致。名前探しの最初の一手。
  6. backtrace の既定 depth は 6。大きなバイナリのハブ関数では --depth 12 --max-paths 1024 が必要なことが多い。
  7. tags --filter はカノニカルな kind スラッグ（anti_debug、Anti-Debug ではない）。
